System Architecture

Factory Inventory Management System

A demo full-stack inventory and operations dashboard. A Vue 3 single-page application talks to a Python FastAPI service over plain HTTP; the service reads its entire dataset from JSON files into memory at startup. There is no database and no persistence layer.

Data flow

The dominant path through the system: a user changes a filter, and every view re-derives from the server response.

  1. User changes a filter
    FilterBar.vue binds four selects — Time Period, Warehouse, Category, Order Status — to refs held in useFilters.js.
  2. Shared singleton state updates
    The refs are declared at module scope, outside the exported function, so every component importing useFilters() reads the same four values. That is what makes one filter bar drive six views without a store.
  3. Views watch and refetch
    Each view calls getCurrentFilters(), which maps UI names onto API names — selectedLocation becomes warehouse, selectedPeriod becomes month.
  4. api.js builds the query string
    Any filter still set to 'all' is omitted rather than sent, so an unfiltered request carries no query params at all.
  5. FastAPI filters in memory
    apply_filters() chains warehouse, category and status list comprehensions (case-insensitive on the latter two). filter_by_month() then matches order_date against either a literal month or a quarter expanded through QUARTER_MAP.
  6. Pydantic validates the response
    response_model coerces and checks each record on the way out. A JSON data file that drifts from its model fails here, at the boundary, rather than in the browser.
  7. Vue stores raw, derives the rest
    Responses land in refs such as allOrders and inventoryItems. Every KPI, chart series and table slice is a computed over those refs — never a second copy of the data.

Observations

Points worth knowing before changing anything — verified against the running services.

The client calls six endpoints the server does not implement

api.js defines task CRUD (GET POST DELETE PATCH on /api/tasks) and purchase-order creation and lookup on /api/purchase-orders. None of these routes exist in main.py — all return 404 against the live server. The failures are caught and logged in App.vue, so the Tasks modal silently falls back to the hard-coded list in useAuth.js and the feature appears to work.

Reports are recomputed per request

/reports/quarterly and /reports/monthly-trends iterate all 250 orders on every call. Fine at this size; the first thing to cache if the dataset grows.

Quarter matching is string containment, not date parsing

Both the filter helper and the quarterly report classify orders with substring checks such as '2025-01' in order_date. That keeps the code dependency-free but hard-codes 2025 — data from another year silently drops out of every quarterly result.

No authentication, and CORS is open

allow_origins=["*"] with allow_credentials=True, and useAuth returns a fixed user. Documented as intentional for a local demo, but it means the API is unauthenticated to anything that can reach port 8001.
